6 Broken Link Hijacking (BLH)

Vulnerability

Broken Link Hijacking (BLH) is a sweet spot for bug hunters.

When external links on a website point to domains that are no longer registered, you can potentially find a security issue.

Note: BLH occurs when you can register a domain that a target website links to, allowing you to serve malicious content to their users.

11 Password Reset Vulnerabilities (1-10)

Authentication

Password reset functionality is often a weak link. Try these hacks: 

1. Test if sessions expire after 2FA and if other sessions can change passwords
2. Check for session fixation issues
3. Verify if reset password links are sent over secure protocols (HTTPS)
4. Test cookie persistence by copying login cookies, logging out, and pasting them
5. Check for broken authentication and session management
6. Test for session hijacking due to improper sanitization
7. Verify if server-side credentials are stored in plaintext
8. Test if the same email or username can be used across different accounts
9. Check if reset password tokens can be used via third parties
10. Verify if reset links expire and if tokens can be reused

15 Dangerous HTTP Methods

Web

HTTP method hacking can lead to some sweet findings. Try these techniques:

1. Use the OPTIONS method to identify which methods are enabled on the server

2. Test the following methods for potential vulnerabilities:

Can potentially store enclosed entities on the server

GET

Retrieves requested information

HEAD

Retrieves only header information

POST

Sends requests with enclosed data entities

DELETE

Removes resources from the server

TRACE

Used for diagnosing communication

CONNECT

Creates tunnels with proxies

19 Session Management Vulnerabilities

Authentication

Session management vulnerabilities can lead to account takeovers - the holy grail of bug hunting:

1. Test for guessable session IDs using tools like Hash-Identifier

2. Check if cookies store sensitive information like session IDs, user data, or plaintext passwords

3. Examine localStorage/sessionStorage for sensitive data

4. Test session invalidation:

  • Log into the website
  • Capture requests (e.g., profile edit) using Burp Proxy
  • Log out from the website
  • Replay captured requests and observe if they still work

Subdomain Recon Roadmap

Interactive

Full commands block: subdomain enumeration, merging, probing, takeover checks, tech fingerprinting, additional recon and a quick workflow.

bash
Step 1
# 1. Identify the target
# Start by choosing the main domain you want to test, e.g. example.com.
# You usually get this from bug bounty platforms such as HackerOne or Bugcrowd.
# Make sure you review the scope and authorization before proceeding.

echo "Target: example.com"
echo "Scope: *.example.com"
echo "Authorization: Yes (via HackerOne/Bugcrowd)"

Always verify the scope and authorization before starting any reconnaissance.

bash
Step 2
# 2. Discover subdomains
# Use a mix of tools to maximize your coverage. Different tools pull from different sources.

subfinder -d example.com -o subfinder.txt
assetfinder --subs-only example.com > assetfinder.txt
amass enum -passive -d example.com -o amass.txt
dnsx -w /usr/share/wordlists/amass/subdomains-top1mil-110000.txt -d example.com -silent -o dnsx.txt
curl -s "https://crt.sh/?q=%25.example.com&output=json" | jq -r '.[].name_value' | sort -u > crtsh.txt
cat subfinder.txt assetfinder.txt amass.txt crtsh.txt dnsx.txt > all_subs.txt
sort -u all_subs.txt > unique_subs.txt
httpx -l unique_subs.txt -o live_subs.txt
cat unique_subs.txt | httprobe > live_subs.txt
subjack -w unique_subs.txt -t 50 -timeout 30 -o takeover.txt -ssl
cat live_subs.txt | katana > urls.txt
cat live_subs.txt | waybackurls >> urls.txt 
cat live_subs.txt | gau >> urls.txt
sort -u urls.txt > unique_urls.txt
httpx -l unique_urls.txt -o live_urls.txt
dnsx -l subs.txt -a -resp
naabu -iL live_subs.txt -o ports.txt
bash
Step 3
# 3. Merge and remove duplicates
# Combine results from all tools into one file and sort to keep only unique subdomains.

cat subfinder.txt assetfinder.txt amass.txt crtsh.txt dnsx.txt > all_subs.txt
sort -u all_subs.txt > unique_subs.txt

# Count results
echo "Total unique subdomains found: $(wc -l < unique_subs.txt)"
bash
Step 4
# 4. Check for live subdomains
# Test which subdomains respond over HTTP/HTTPS. This helps focus on active assets only.

httpx -l unique_subs.txt -o live_subs.txt
# alternative:
cat unique_subs.txt | httprobe > live_subs.txt

# With more details
httpx -l unique_subs.txt -sc -title -td -o live_detailed.txt
bash
Step 5
# 5. Check for subdomain takeover
# Look for dangling DNS records that may allow an attacker to claim the subdomain.

subjack -w unique_subs.txt -t 50 -timeout 30 -o takeover.txt -ssl
# alternative tools:
subzy run --targets=unique_subs.txt
nuclei -t takeovers/ -l unique_subs.txt
bash
Step 6
# 6. Detect technologies and vulnerabilities
# Fingerprint technologies used by the site and run port/service scans.

whatweb -i live_subs.txt -o whatweb.txt
nmap -iL live_subs.txt -sV -T4 -oN nmap_scan.txt
npx wappalyzer https://sub.example.com

# BuiltWith for tech profiling
builtwith -v https://example.com
bash
Step 7
# 7. Additional recon tools
# Get historical URLs, parameters and more surface area for fuzzing.

cat live_subs.txt | katana > urls.txt
cat live_subs.txt | waybackurls >> urls.txt 
cat live_subs.txt | gau >> urls.txt
sort -u urls.txt > unique_urls.txt
httpx -l unique_urls.txt -o live_urls.txt
dnsx -l subs.txt -a -resp
naabu -iL live_subs.txt -o ports.txt
bash
Step 8
# 8. Quick example workflow
# Run a simple chain to quickly enumerate, probe, and scan.

subfinder -d target.com -o s1.txt
assetfinder --subs-only target.com > s2.txt
dnsx -w /usr/share/wordlists/amass/subdomains-top1mil-110000.txt -d target.com -silent -o s3.txt

cat s*.txt > all.txt
sort -u all.txt > subs.txt

httpx -l subs.txt -o live.txt
subjack -w subs.txt -t 50 -o takeover.txt -ssl
whatweb -i live.txt -o whatweb.txt

# Additional recon
cat live.txt | waybackurls > urls.txt
cat live.txt | gau >> urls.txt
sort -u urls.txt > unique_urls.txt

Keep learning. Every subdomain is a potential entry point.

XSS — DISCOVER & PREPARE

# Discover reflected XSS (passive + active)
(
  echo "shop.turbo-eg.com" | gau
  echo "https://shop.turbo-eg.com" | katana -silent -jc -kf -ps
) | \
gf xss | \
uro | \
gxss | \
kxss | \
tee xss_output.txt

# Extract vulnerable URLs and prepare for fuzzing
grep -oP 'URL: \K\S+' xss_output.txt \
| sed 's/=[^&]*/=FUZZ/g' \
| sort -u > final.txt

Note: This section performs XSS discovery and post-processing in one workflow.
Results are candidates only — always verify manually in the browser or Burp.
Run active crawling only on in-scope targets.

JS & Parameters Discovery (Detailed)

JavaScript
# extract js links
awk '/\.js(\?|$)/ {print}' urls.txt > js_links.txt

# quick download (be careful; obey robots & scope)
mkdir -p js && cd js && wget -q -i ../js_links.txt

# extract parameters
cat urls.txt | grep -oP '\?\K[^ ]+' | sort -u > params.txt

# search for tokens in JS (example)
grep -RInE "api_key|token|access|client_secret" js/ || true

# extract endpoints from JS files
cat js_links.txt | while read url; do
  echo "=== $url ==="
  curl -s "$url" | grep -oE "['\"](https?:)?//[^'\"?#]+" | sort -u
done > js_endpoints.txt

Always respect robots.txt and scope. Do not exfiltrate or misuse data.

CORS Quick Check

Security
# quick automated CORS finder (parallel)
cat unique_urls.txt | xargs -P30 -I% sh -c 'curl -s -I -H "Origin: https://evil.com" "%" | grep -iE "access-control-allow-origin|access-control-allow-credentials" && echo "[POTENTIAL CORS] %"'

# alternative with more headers
cat live_urls.txt | while read url; do
  response=$(curl -s -I -H "Origin: https://attacker.com" "$url")
  if echo "$response" | grep -qi "access-control-allow-origin: https://attacker.com"; then
    if echo "$response" | grep -qi "access-control-allow-credentials: true"; then
      echo "[CRITICAL CORS] $url - Credentials allowed"
    else
      echo "[CORS FOUND] $url"
    fi
  fi
done

Parallel requests increase load — throttle to stay in-scope.
